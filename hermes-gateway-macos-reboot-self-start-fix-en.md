# Hermes Gateway macOS Auto-Start Fix — Full Story

> Author: Corton Kwok
> Environment: macOS 26.5.2 (Sequoia), Hermes Agent v0.17.0-cn.5
> Hermes Issue: #23387, PR: #62223

---

## Problem

After macOS reboot, Hermes Gateway (launchd service) fails to auto-start. Despite `RunAtLoad=true` and `KeepAlive=true` in the plist, the Gateway process does not appear after reboot.

### Impact

- Gateway offline → messaging platforms (WeChat, etc.) disconnected
- Cron scheduler (running inside Gateway) offline → watchdog scripts not triggered
- Requires manual `hermes gateway start` to recover

## Root Cause

Two independent but compounding issues:

### Issue A: `LimitLoadToSessionType` causes silent load failure on macOS 26.5.2

The plist template generated by `hermes gateway install` includes:

```xml
<key>LimitLoadToSessionType</key>
<array>
    <string>Aqua</string>
    <string>Background</string>
</array>
```

On macOS 26.5.2, launchd silently fails to auto-load plists with this key during login. Even manual `launchctl bootstrap` or `launchctl load -w` returns exit code 5 (EIO).

### Issue B: `launchctl bootstrap` is broken on macOS 26+

Hermes uses `launchctl bootstrap` in 6 code paths to load service definitions:
- `launchd_install()` — first-time install
- `launchd_start()` — start service (plist missing + reload paths)
- `launchd_restart()` — restart service
- `refresh_launchd_plist_if_needed()` — inline reload + deferred shell script

On macOS 26+, `launchctl bootstrap` returns exit code 5 (EIO) in all scenarios. Hermes has a fallback (`_launchd_fallback_to_detached`) but it only triggers during manual `start`/`install` — the auto-start on reboot has no Hermes code running to trigger it.

The alternative `launchctl load` + `launchctl enable` works on all macOS versions.

## PR Review and Iteration

After submitting PR #62223, community maintainer **teknium1** (Hermes Agent author) raised 3 issues via automated code review:

### Issue 1: Unconditional `LimitLoadToSessionType` removal breaks Background sessions

**Original fix**: Delete the key entirely.
**Review finding**: Commit `360630733` intentionally added Background session support after the #23387 investigation. Removing it entirely would prevent plist loading in SSH / non-Aqua login sessions.
**Fix**: Version-conditional output — `_limit_load_section()` omits the key on macOS 26+, preserves `[Aqua, Background]` on < 26.

### Issue 2: Fix only covered `_launchctl_bootstrap`, missed other call sites

**Original fix**: Only modified the internal logic of `_launchctl_bootstrap()`.
**Review finding**: The deferred reload script (shell string in `refresh_launchd_plist_if_needed`) and restart recovery (`launchd_restart`'s unloaded branch) call `launchctl bootstrap` directly, bypassing `_launchctl_bootstrap()`.
**Fix**: Both sites now use version-aware branches — `launchctl load/enable` on macOS 26+, `bootstrap` on < 26.

### Issue 3: No tests added

**Original fix**: Changed plist output and command selection with 0 test changes.
**Review finding**: Needs version-aware regression tests covering both plist output and macOS 26+ load/enable paths.
**Fix**: Added 8 new tests:

| Test | Coverage |
|------|----------|
| `test_launchd_plist_omits_limit_load_on_macos_26` | macOS 26+ plist lacks `LimitLoadToSessionType` |
| `TestLaunchctlBootstrapMacOs26` (3 tests) | `load/enable` normal / load failure / enable failure |
| `test_launchd_start_reloads_with_load_enable_on_macos_26` | `launchd_start()` kickstart fail→load/enable→kickstart |
| `test_launchd_restart_reloads_using_launchctl_bootstrap_on_macos_26` | `launchd_restart()` unloaded branch |
| `TestRetryLaunchctlBootstrapUntilRegisteredMacOs26` (2 tests) | retry loop registration verify + TimeoutExpired retry |

Also added `autouse fixture` to existing test classes (`TestLaunchctlBootstrapEioRetry`, `TestRetryLaunchctlBootstrapUntilRegistered`) locking them to macOS < 26 path, ensuring correct test execution on macOS 26.5.2 hosts.

## Final Solution Architecture

```
launchctl bootstrap call sites (6 total)
│
├── _launchctl_bootstrap() ← unified entry
│   ├── macOS < 26: launchctl bootstrap (with EIO stale-label recovery)
│   └── macOS 26+:  launchctl load + launchctl enable
│
├── launchd_install()        → _launchctl_bootstrap() ✅
├── launchd_start() (2)      → _launchctl_bootstrap() ✅
├── launchd_restart()        → 26+: _launchctl_bootstrap() / <26: bootout+bootstrap ✅
└── refresh_launchd_plist_if_needed()
    ├── inline path           → _retry_→_launchctl_bootstrap() ✅
    └── deferred shell script → version-aware shell string ✅
```

### Residual Risks

- **Hardcoded version detection**: `_is_macos_26_or_later()` matches major version via `platform.mac_ver()`. Conservative — macOS 27+ even if `bootstrap` is fixed, using `load/enable` is harmless. A runtime probe (try `bootstrap`, fallback on exit 5) is cleaner but interacts poorly with static plist generation (`_limit_load_section()`).
- **Stale-label recovery**: macOS 26+ `load/enable` path lacks bootout-before-load. Practically all callers either bootout first (`launchd_restart`, `refresh`) or are in fresh-install scenarios (`launchd_install`, `launchd_start`), so this is unlikely to trigger. Could add `launchctl bootout 2>/dev/null` before load as safety margin.
